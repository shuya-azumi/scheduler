## 0. 一言サマリ
人材紹介（エージェント）が関わる採用で、**面接の日程調整に時間がかかり、その間に求職者の温度感が下がって辞退に至る**という課題を解決したい。ホワイトカラー大手ではなく、**介護・クリニックなど中小・分散型市場**を狙う。Google等のデジタルカレンダーを使っていない事業所が多い、という前提で設計する。

## 1. 背景・きっかけ
- 作成者の友人が **人材紹介サービス**と**介護事業所**の両方を運営。
- 両者に共通する最大の悩みが「**面接の日程調整に非常に時間がかかる**」こと。本人いわく「ここが一番難しい」。
- 作成者はもともと Firebase でチャットアプリを学習中。その技術を実務に活かせる題材として本件を検討している。

## 2. 解決したい課題（構造）
**三者が介在することで調整が伝言ゲーム化する：**
1. 求職者 ⇄ 人材紹介エージェント で日程調整
2. その内容を、エージェントが事業所側の面接官へ伝えて再調整（＝伝言ゲーム／電子鳩）
3. このラリーが何度も往復する

**ラリーが長引く → 求職者の温度感が下がる → 辞退。** エージェントにとって辞退は成功報酬（数十万円規模）の喪失に直結する。
つまりこれは「便利機能」ではなく**売上ロスの直接対策**＝払う理由が明確な課題。

## 3. なぜ今まで解けていないか（診断）
| 仮説 | 評価 |
|---|---|
| ① カレンダーの権限・機密情報を業者に公開するリスク | **ホワイトカラー側の理由。かつ概ね解決済み**（Spir/TimeRex/Jicoo が「予定の中身を見せず空き時間だけ共有」を実現）。ここに空き地は少ない |
| ② 小規模の介護/クリニックは、そもそもデジタルカレンダーを使っていない | **★本丸。未対応の本当の理由。** 連携元データが無いのでGoogle連携型ツールは原理的に使えない。手帳・ホワイトボード・LINE・口頭が現実（**要・実態調査**） |
| ③ 既存ツールは「1対1セルフ予約（Calendlyモデル）」前提 | 人材紹介は**エージェントが司令塔の“3者オーケストレーション”**（複数の求職者×複数の事業所を同時にさばく／空き収集→重なり探索→両者確定→リスケ吸収）。このUXは手薄 |

**設計の核心**：**「事業所はカレンダーを持っていない」前提で、エージェントを主役に設計する。** 面接官に「カレンダーを使え」と要求しない。代わりに**極限まで軽い空き枠入力**（候補日を提示→LINE/ボタンで1タップ返信 等）で吸い上げる。

## 4. 攻め筋（MVPの“楔”は2案）
求職者が冷める2大ポイント＝「面接前の調整待ち」と「面接後の結果待ち」。それぞれに対応する。

| | 楔A：3者 日程調整コーディネーター | 楔B：面接後 即時フィードバック |
|---|---|---|
| 狙う痛み | 面接“前”の往復ラリー（友人が言う最大の痛み） | 面接“後”の結果待ち（もう一つの冷却点） |
| 価値 | 大きい | 中〜大 |
| 作りやすさ | 中（空き枠収集UXが肝） | **小** |
| カレンダー依存 | 設計次第で回避可能 | **ゼロ** |
| 導入摩擦 | 中（3者を巻き込む） | **低（面接官が1人でタップするだけ）** |

**楔Bの具体イメージ**（作成者の構想）：
- 面接終了後、面接官が **〈進める／見送る／迷う〉をボタンで選択**
- 続けて**選択式の評価**を入力
- → エージェントや社内の他担当者へ**自動でメール通知**＝選考結果を即時共有

**戦略仮説**：まず**楔B**で小さく価値を見せ（カレンダー問題を踏まない・即日使える）、信頼ができてから**楔A**（本丸）に入る“トロイの木馬”。ただし「最大の痛みから殴る＝Aから」も選択肢。**ここは全体設計で決めたい論点。**

**市場の方針**：ホワイトカラー（多段選考・既存ATS/Outlook連携・競合多数）は複雑なので避ける。**中小・分散型（介護/クリニック/ブルーカラー寄り、多くは面接1回）**に絞る。

## 5. 想定ユーザー／課金者（論点）
| 候補 | 妥当性 |
|---|---|
| 人材紹介エージェント（友人） | **第一候補。** 痛みの当事者・辞退＝報酬喪失・3者を動かす司令塔＝最も払う動機が強い |
| 事業所（介護/クリニック） | 導入摩擦は高いが、直接採用にも使える可能性 |
| 求職者 | 温度感が下がる本人だが、課金者にはなりにくい |

→ **第一候補はエージェント**。ただし全体設計で確定する。

## 6. 未検証＝先に潰すべきリサーチ（友人へのヒアリング項目）
1. 事業所の面接官は**何で予定管理しているか**（Googleカレンダー／LINE／紙・ホワイトボード／なし）← アーキテクチャ全体が決まる最重要質問
2. 1件の採用で**日程調整は平均何往復・何日**かかるか。遅延で年に何人逃したか（＝ROIの数字）
3. **リスケ（決定済み面接の再調整）**の頻度（地獄になりがち）
4. 今は何で調整しているか（電話/メール/LINE/Spir等を試して何がダメだったか）
5. **誰が払うか**（エージェント＝友人か、事業所か）
6. 友人の紹介会社は**月に何件の面接**を回しているか（市場規模の当たり）

## 7. 技術前提（実装フェーズで Claude Code に戻る際の参考）
- 基盤は **Firebase** を想定。
- 作成者の現在地：Firebase **Realtime Database** ＋ jQuery でリアルタイムチャットの最小実装は可能（push/set/onChildAdded）。**認証なし**の段階。
- 本サービスで新たに要る想定技術：
  - **Firebase Authentication**（求職者・エージェント・事業所の3ロール）
  - **Firestore**（予定・評価など構造化＆検索が必要なデータ。チャット用のRealtime DBより向く）
  - **リアルタイム更新**（学習済みの考え方が活用可能）
  - **メール自動送信**（楔Bの肝。Cloud Functions＋メールAPI、最小ならフォーム連携でも可）
- 結論：**楔BのMVPは現スキル＋αで射程内**。実在の課題×実在の最初の検証パートナー（友人）で作れる理想的な題材。

## 9. 企画チャットで決めたいこと（全体設計の論点）
1. **MVPの楔をA/Bどちらにするか**（または統合した「面接オペレーション」ツールとして一体設計するか）
2. **主役＝課金者を誰に置くか**（エージェント／事業所）
3. **マネタイズ**（月額SaaS／成果報酬連動／従量 等）と価格帯
4. **ビーチヘッド市場の確定**（介護 or クリニック or その他中小）と、横展開の順番
5. **競合との差別化**（Spir/TimeRex/Jicoo 等のセルフ予約ツール、及び中小向けATSに対する勝ち筋）
6. **MVPのスコープと検証KPI**（何を作り、何が起きたら「当たり」と判断するか）
7. **リスク**（事業所のデジタル非習熟、個人情報＝求職者データの扱い、エージェントの業法＝職業安定法まわり）

> この7点を企画チャットで整理 → 結果を持って Claude Code に戻り、選んだ楔の MVP を Firebase で実装する。
